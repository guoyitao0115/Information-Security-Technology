## 三、选题理由

本组拟选择论文 Characterizing the Security of GitHub CI Workflows。该论文发表于 USENIX Security 2022，属于信息安全领域的高水平会议论文，时间和会议等级均符合课程要求。

该论文研究 GitHub Actions 工作流中的安全问题。GitHub Actions 是当前开源项目中广泛使用的 CI/CD 自动化工具，开发者可以通过 YAML 工作流文件配置构建、测试、部署等任务。由于工作流通常拥有代码仓库权限、Secret 访问能力和自动执行能力，一旦配置不当，就可能造成供应链攻击、权限滥用、敏感信息泄露或恶意代码执行。

选择该论文的主要原因如下：

1. 研究问题清晰，容易围绕“GitHub Actions 工作流安全风险检测”展开报告。
2. 实验环境可控，不需要 GPU、大规模训练数据或复杂硬件。
3. 适合进行工程复现，可以实现一个简化版工作流安全扫描器。
4. 适合提出改进，例如风险评分、修复建议、规则扩展和结果可视化。
5. 实验结果容易用表格、图表和运行截图呈现，符合课程报告对实验数据和结果分析的要求。

## 四、论文核心内容概述

论文围绕 GitHub Actions 工作流安全展开，主要关注开源仓库中 CI/CD 配置文件可能引入的安全风险。其核心思想是分析 GitHub Actions workflow 的配置结构、权限模型、触发条件和第三方 Action 使用情况，从而识别潜在的不安全配置。

论文中的核心方案可以概括为：

1. 收集 GitHub 仓库中的 workflow 配置文件。
2. 解析 YAML 格式的 GitHub Actions 工作流。
3. 提取触发器、权限、作业步骤、第三方 Action、Secret 使用方式等关键信息。
4. 根据安全规则判断 workflow 是否存在风险。
5. 对检测结果进行统计分析，归纳 GitHub Actions 生态中的常见安全问题。

本课程项目不会完整复现论文的大规模数据采集系统，而是围绕其核心安全检测思想进行可控复现，并在此基础上增加改进模块。

## 五、拟复现方案

### 5.1 复现目标

复现目标是实现一个基础版 GitHub Actions 工作流安全检测工具，对 `.github/workflows/*.yml` 或 `.github/workflows/*.yaml` 文件进行静态扫描，识别常见安全风险。

### 5.2 基础检测规则

基础版检测器计划实现以下规则：

1. 检测是否使用 `pull_request_target` 触发器。
2. 检测是否缺少显式 `permissions` 配置。
3. 检测是否存在过高权限，例如 `contents: write`、`actions: write`、`packages: write`。
4. 检测第三方 Action 是否未固定到 commit SHA。
5. 检测 workflow 步骤中是否存在危险命令模式，例如直接执行外部脚本。
6. 检测是否存在可疑的 Secret 或环境变量使用方式。

### 5.3 工具输入与输出

输入：

- 单个 workflow 文件
- 包含多个 workflow 文件的项目目录
- 自建测试样例目录

输出：

- 控制台扫描结果
- JSON 结构化结果
- CSV 汇总结果
- 风险类型统计表
- 运行截图

## 六、拟改进方案

在基础复现的规则检测能力之上，本项目计划实现一个改进版检测器。改进重点不是单纯增加规则数量，而是增强检测结果的解释性、可比较性和可用于修复的价值。

### 6.1 风险评分模型

为每条检测规则设置风险权重，并根据命中的风险项计算 workflow 的综合风险分数。

拟定风险等级如下：

| 风险等级 | 分数范围 | 含义 |
|---|---:|---|
| 低风险 | 1-3 | 存在一般配置问题，建议优化 |
| 中风险 | 4-7 | 存在可能被利用的安全隐患，应尽快修复 |
| 高风险 | 8 及以上 | 存在明显危险配置，可能导致权限滥用或供应链攻击 |

### 6.2 修复建议生成

改进版检测器将为每种风险生成对应修复建议。例如：

- 对缺少 `permissions` 的 workflow，建议显式设置最小权限。
- 对使用 `pull_request_target` 的 workflow，建议谨慎处理外部 PR 输入，避免 checkout 未可信代码。
- 对未固定版本的第三方 Action，建议固定到完整 commit SHA。
- 对危险 shell 命令，建议改为可信脚本、校验下载源或减少动态执行。

### 6.3 结果可视化

改进版工具将生成实验图表，包括：

1. 风险类型分布柱状图。
2. 基础方案与改进方案检测能力对比图。
3. 不同样例 workflow 的风险评分对比图。
4. 实验流程图或工具架构图。

这些图表将插入最终实验报告，用于支撑结果分析。

## 七、实验设计

### 7.1 实验环境

实验环境计划如下：

| 项目 | 配置 |
|---|---|
| 操作系统 | Windows |
| 编程语言 | Python 3 |
| 主要依赖 | PyYAML、matplotlib、pandas |
| 输入文件 | GitHub Actions workflow YAML 文件 |
| 输出文件 | JSON、CSV、PNG 图表、运行截图 |

### 7.2 实验数据

实验数据分为两类：

1. 自建样例 workflow：构造安全样例、低风险样例、中风险样例和高风险样例，用于验证检测器规则是否正确。
2. 公开仓库 workflow：选取若干 GitHub 开源项目中的 workflow 文件，作为更接近真实场景的测试数据。

如果公开仓库数据获取受到网络或时间限制，最终实验仍可基于自建样例和少量公开样例完成。自建样例会覆盖主要风险规则，保证实验可复现。

### 7.3 实验步骤

1. 准备 workflow 样例文件。
2. 运行基础版检测器，记录规则命中结果。
3. 运行改进版检测器，记录风险评分、风险等级和修复建议。
4. 汇总结果到 CSV。
5. 生成统计图表。
6. 对典型高风险样例进行案例分析。
7. 比较基础方案与改进方案在结果表达、风险区分和修复指导方面的差异。

### 7.4 实验评价指标

计划采用以下指标评价改进效果：

| 指标 | 说明 |
|---|---|
| 检测规则覆盖数 | 工具能够识别的风险类型数量 |
| 风险命中数量 | 每个样例中检测到的风险项数量 |
| 风险评分 | 改进模型对 workflow 风险程度的量化结果 |
| 风险等级 | 将扫描结果划分为低、中、高风险 |
| 修复建议完整性 | 是否能为命中风险提供具体修复方向 |
| 结果可解释性 | 输出是否便于理解和写入报告分析 |
